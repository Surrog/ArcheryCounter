## Project Overview

ArcheryCounter is a React Native app that detects arrows in archery target photos and scores them automatically. The detection logic is written in C++17 with OpenCV and exposed to React Native via native modules for iOS and Android.

## Repository Structure

```
native/       C++ algorithm source + original desktop CLI build
cpp/          Platform-independent C++ wrapper (processImageFile entry point)
ios/          React Native iOS project (Xcode workspace + native module bridge)
android/      React Native Android project (Gradle + JNI bridge)
src/          React Native JS/TS source (screens, components, hooks)
docs/         Architecture and migration documentation
```

## C++ Desktop Build (native/)

OpenCV must be installed. On macOS: `brew install opencv`. On Linux: `sudo apt-get install libopencv-dev`.

```bash
mkdir native/build && cd native/build
cmake ..
cmake --build . -- -j$(nproc)
```

On Linux CI, the build uses `scan-build` (clang static analyzer):
```bash
scan-build cmake --build . -- -j2
```

`native/sources.hpp` is auto-generated by CMake from `native/sources.hpp.in` ‚Äî do not edit it directly.

## Architecture

### C++ Pipeline (`native/`)

**Stage 1 ‚Äî Target detection** (`ellipses` namespace in `ellispes_approach.*`):
- `ellipses::find_target()` returns `std::array<cv::RotatedRect, 10>` ‚Äî the 10 concentric ring ellipses
- Applies `filter_image()` (5 HSV colour filters: yellow/red/blue/black/white), detects ellipses from the first 3 in parallel via `std::async`, discards the weakest, then linearly interpolates all 10 rings
- `compute_target_ring_mask()` in `main.cpp` converts ellipses into filled binary masks

**Stage 2 ‚Äî Arrow detection** (`NN` namespace in `NN_approach.hpp`):
- `NN::find_arrows()` ‚Äî currently a stub returning empty results
- Scoring: `get_arrow_point()` walks masks outermost‚Üíinnermost; returns 0 = miss, 10 = bullseye

**Supporting modules:**
- `pretreatment.*` ‚Äî grayscale, blur, erode/dilate before detection
- `colour_filters.*` ‚Äî HSV-space filters; `filter_image()` returns all 5 as `std::array<cv::Mat, 5>`
- `math_utils.hpp` ‚Äî `euclideanDistance`, `center_of_mass`, `mean_distance`
- `hough_approach.*`, `histo_approach.*`, `keypoint_approach.*` ‚Äî alternative approaches, not used in main pipeline

### Native Module Bridge

`cpp/ArcheryCounterModule.{hpp,cpp}` is the single platform-independent entry point:
```cpp
ArcheryResult processImageFile(const std::string& imagePath);
```
- iOS: `ios/ArcheryCounterBridge/RCTArcheryCounter.mm` (Obj-C++) calls this, exposes as `NativeModules.ArcheryCounter`
- Android: `android/app/src/main/jni/ArcheryCounterJNI.cpp` (JNI) calls this, `ArcheryCounterModule.java` wraps it

### React Native JS (`src/`)

- `NativeArcheryCounter.ts` ‚Äî typed bridge to `NativeModules.ArcheryCounter`
- `useArcheryScorer.ts` ‚Äî hook: image picking ‚Üí `processImage()` ‚Üí state
- `components/RingOverlay.tsx` ‚Äî SVG ellipse overlay with letterbox-aware coordinate transform
- `screens/HomeScreen.tsx` ‚Äî main screen

## Key Design Notes

- The 10-ring mask array: index 0 = outermost ring, index 9 = bullseye
- `NN_approach.hpp` is header-only; `find_arrows` is a placeholder stub
- `std::async` is used in both `colour_filters.cpp` and `ellispes_approach.cpp` ‚Äî on Android this requires `ANDROID_STL=c++_shared` to avoid ODR violations with OpenCV's libc++ usage
- iOS uses `OpenCV-Dynamic-Framework` (~> 4.10) via CocoaPods ‚Äî not `OpenCV2`, which lacks simulator slices
